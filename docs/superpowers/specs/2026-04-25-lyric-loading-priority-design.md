# 歌词加载优先级设计

## 概述

统一后端扫描和前端播放时的歌词加载逻辑，实现一致的优先级策略。

## 优先级顺序

```
1. 在线匹配缓存歌词（用户手动在线匹配后保存的歌词）
   ↓ 如果为空或不存在
2. 同名 .lrc 文件（必须非空，至少包含一个非空白字符）
   ↓ 如果不存在或为空
3. 音频文件内嵌歌词（ID3 标签中的 Lyrics 字段）
   ↓ 如果不存在或为空
4. 无歌词
```

## 空文件处理规则

- 文件存在但内容为空（0 字节）→ 视为无歌词，继续尝试下一优先级
- 文件存在但只包含空白字符（空格、换行等）→ 视为无歌词，继续尝试下一优先级
- 文件存在且包含有效内容 → 使用该歌词

## 实现位置

### 后端（Rust）- `commands.rs`

修改 `parse_audio_file` 函数中的歌词扫描逻辑：

```rust
// 1. 检查在线匹配缓存歌词（通过缓存服务获取）
// 2. 读取同名 .lrc 文件（检查非空）
// 3. 读取音频文件内嵌歌词
```

### 前端（Vue）- `App.vue`

修改 `playSong` 函数中的歌词解析逻辑：

```typescript
// 1. 检查在线匹配缓存歌词
// 2. 使用歌曲对象中的 lyric 字段（来自后端扫描）
// 3. 动态读取同名 .lrc 文件（检查非空）
```

## 缓存歌词存储

在线匹配后的歌词保存到 localStorage，使用歌曲 ID 作为键：

```typescript
// 存储格式
{
  "song_id_lyric_cache": "歌词内容"
}
```

## 验证规则

歌词内容有效性检查：
- 长度 > 0
- 去除空白字符后长度 > 0
- 至少包含一个非空白字符

## 日志输出

每个优先级步骤都输出调试日志，方便追踪歌词来源：

```
[歌词加载] 优先级 1: 检查在线匹配缓存...
[歌词加载] 优先级 2: 检查同名.lrc文件...
[歌词加载] 优先级 3: 检查内嵌歌词...
[歌词加载] 最终结果: 使用来源 X，长度 Y
```
